### 1.1 F-1 に対する横断要件

デコード非同期化は単に処理を別スレッドへ移すだけでなく、次を満たすこと。本節は**満たすべき結果（何を）**を
規定し、実現方式（どう）は v1.0 詳細設計書 §4.1 に委ねる。

> なぜ機構を書かないか: 要求定義は「表示が遅れない／RAM が有界」といった**結果**を縛る層であり、
> 二段キューや RAM 予約などの**実装機構**は詳細設計の領分。要求側に機構を書くと設計の自由度を不必要に
> 拘束し、後の簡素化の妨げになる。

- **表示対象のデコードを先読みより優先する。** 表示枠の画像が、先読み候補に待たされて表示が目に見えて
  遅れないこと。先読み要求が既に待機中またはキュー満杯でも、表示要求を受け付ける容量と実行優先度を確保する。
- **デコード済み画像（CPU 側 RGBA）の RAM 使用量を有界に保つ。** F-8 により巨大画像はスライドショー対象から
  除外される（寸法上限を超える画像は退避）ため、デコードする 1 枚は寸法上限以下に収まる。これと先読み枚数の
  上限により RAM を有界に保つ（全段階を跨ぐバイト予約・例外枠は要しない。§8 / 詳細設計 §4.1）。
- **非同期結果を要求世代または要求IDで識別する。** 同じパスが再登録・再作成・再要求された場合も、
  古い成功／失敗結果を現在の候補へ適用しないこと。
- 要求後に同じパスのファイル内容が外部で差し替えられた場合も、要求時のファイル情報と適用時の
  ファイル情報を照合し、古い成功／失敗／巨大画像判定を新しいファイルへ適用しないこと。
- 現在表示する 1〜2 枚は VRAM 上限より優先する。3 枚目以降の先読みだけを VRAM 上限で停止する。
- 非同期の失敗・外部消失で候補列が短縮された場合も、F-4 の共通状態遷移を必ず通し、
  `cursor >= play_order.len()` かつ窓空で停止しないこと。
- F-3〜F-8 を確定仕様として扱う。過去の設計メモや 前身 文書と競合する場合は、
  本要求定義書と v1.0 詳細設計書を優先する。

## 2. F-2：メニュー表示中の自動送り一時停止（残り時間から再開）

### 2.1 背景

前身 では、画像を右クリックして削除コンテキストメニューを開いている間も
自動送り（advance）のタイマーが進み続ける（`app.rs` の advance 判定は `menu_open` を見ていない）。
そのため、メニューを開いたまま `ADVANCE_INTERVAL_SECS`（15 秒）が経過すると、
右クリックした画像が窓から流れて消え、削除がキャンセルされることがある
（`do_delete_path` → 「対象画像が切り替わったため削除をキャンセルしました」）。

### 2.2 要件

- **コンテキストメニューが開いている間は自動送り（advance）を一時停止する。**
- **再開方式は「残り時間から再開（フリーズ／続きから）」とする。**
  - メニューを開いた時点の「次の送りまでの残り時間」を保持する。
  - メニュー表示中はカウントを進めない（残り時間を凍結）。
  - メニューを閉じたら、凍結した残り時間が経過した時点で送る。
  - 例：残り 3 秒でメニューを開いた場合、閉じてから 3 秒で次へ送られる。

### 2.3 制約・整合

- 再生停止中（`playing == false`）はもともと送らないため、本要件の影響を受けない。
- 定期 rescan（10 秒間隔）はメニュー表示中も継続してよい。
  rescan は表示中ペア（窓 index 0/1）を除去しないため、メニュー操作の妨げにはならない。
- 既存の誤削除防止（パス基準の `menu_target` とキャンセル挙動）は維持する。
  本要件は「そもそも対象が流れにくくする」ものであり、キャンセル機構を置き換えない。
- 再描画スケジュール（`request_repaint_after`）は、一時停止に合わせて
  メニュー中は advance 起因の再描画要求を出さないよう整合させる。
- メニューと管理画面が同時に開き得るため、凍結状態は個別の開閉ではなく
  **`menu_open || settings_open` の総合状態**で管理する。一方を閉じても他方が開いている間は解除しない。
- 停止中にメニューを開き、メニュー内の「再生」を選んだ場合は、凍結前の残り時間ではなく
  **再生操作時点から設定間隔を全て計り直す**。通常再生中のメニュー開閉だけが「残り時間から再開」の対象となる。

### 2.4 受け入れ基準

1. メニューを開くと、上部バーや内部状態上の「次の送りまでの残り時間」が進まない（凍結する）。
2. メニューを閉じると、凍結した残り時間から計時が再開し、その経過後に送られる。
3. メニュー操作中（妥当な操作時間内）は、右クリックした画像が自動送りで窓から消えない。
4. 再生停止中はメニュー開閉に関わらず送られない（既存挙動を維持）。
5. メニューを開いたフレームに送り期限が重なっても送られず、メニューを閉じた同じフレームにも即送られない。

## 3. F-3：Undo（削除取り消し）機能の廃止

### 3.1 背景

前身 の Undo（`Ctrl+Z` ／上部バー「↶ Undo」ボタン）は、直前 1 件の削除のみを取り消せる
単段の機能である（履歴は `quarantine.last: Option<DeleteRecord>` の 1 スロットで、削除のたびに
上書きされる）。実装上、次の問題があり「効いていない／信用できない」と受け取られる。

- **取り消しても画面に即時復元されない。** 成功時は復元ファイルを未表示キューの末尾へ戻して
  再シャッフルするだけで、すぐには再表示されない（`app.rs` の `do_undo`）。
- **成功時の通知が無い。** 画面メッセージは失敗経路でしか出ない。
- **単段のみ。** 2 件以上削除すると古い方は取り消せない。
- **`do_undo` を通した試験が無い。** 回帰で守られていない。

これらは個別に改修も可能だが、v1.0 では Undo 自体を提供しない方針とする。

### 3.2 要件

- **v1.0 では Undo（削除取り消し）機能を廃止する。**
  - `Ctrl+Z` のキーバインドを撤去する。
  - 上部バーの「↶ Undo」ボタンを撤去する。
  - 削除履歴（`quarantine.last` / `DeleteRecord`）と復元経路（`restore`）を撤去する。

### 3.3 制約・整合

- **削除（隔離移動）自体は維持する。** 削除は従来どおり原本を `<入力>/delete` へ移動して行い、
  原本は保全される（読み取り専用方針は不変）。取り消しが必要な場合は、利用者が `delete`
  フォルダから手動で戻す運用とする（アプリ内に取り消し手段は設けない）。
- `delete` への移動先に同名ファイルがある場合は、`name (1).ext`、`name (2).ext` のように
  利用可能な番号を付け、既存ファイルを上書きしない。この衝突回避命名は F-8 の `oversized`
  への移動と共通仕様・共通処理にする。
- v1.0 デコード非同期化 設計提案も本要件に合わせて更新済みであり、非同期処理は Undo の存在を前提にしない。

### 3.4 受け入れ基準

1. `Ctrl+Z` を押しても何も起きない（Undo として機能しない）。
2. 上部バーに「↶ Undo」ボタンが存在しない。
3. 削除は従来どおり `delete` フォルダへ移動され、原本がディスクから消失しない。

## 4. F-4：削除・デコード失敗でサイクル処理が停止する不具合の修復

### 4.1 背景（現象）

スライドショーは「1 サイクルで全画像を重複なく表示し、出し終えたら再シャッフルして繰り返す」。
このサイクル境界（再シャッフル＋`cursor=0`）は、**`advance()` が `cursor` を末尾まで進めた場合にのみ**
発火する設計になっている（`app.rs` の `advance`）。

しかし `cursor` は advance 以外の経路でも末尾に到達し得る。**サイクルの最後に残った 1 枚を
画面上で削除する**と、次の状態になる。

- `do_delete`：表示中の窓を `evict` して **window が空**、`play_order` から当該パスを除去して
  `len` が 1 減る。`cursor` は据え置き。
- 結果、**`cursor == play_order.len()` かつ window 空**（ただし `play_order` は空ではない＝
  表示済みの画像が残っている）。

この状態に陥ると、以後どの経路でも前進できず**停止**する。

- `refill_window`：補充位置 `idx = cursor + window.len() = len` が `len` 以上のため即終了。補充不能。
- `advance()`：`shown = window.len().min(2) == 0` で早期 return し、`cursor += shown` も
  **サイクル境界処理にも到達しない**。
- `rescan`：窓より後ろの走査範囲が空で、追加も無いため復旧しない。

その結果、表示すべき画像が残っているのに画面は「読み込み中...」のまま固まり、
再描画だけが回り続ける（実質フリーズ）。**新規ファイルが追加されるまで自力復帰しない。**

### 4.2 同種の発生経路

- **最後の 1 枚のデコード連続失敗 → 恒久除外**でも同じ状態になる。`handle_decode_fail` が
  当該パスを `play_order` から除去し、`cursor == play_order.len()` に至る。
- **入力フォルダからの外部削除・外部移動**でも発生し得る。`rescan` は表示中の
  `window[0]/[1]` を除去しないが、外部消失の検出経路は `rescan` だけではない。
  `cache::refill_window()` も補充候補の `path.exists()` が偽なら、そのパスを
  `play_order` から即時除去する。この除去後にもサイクル境界判定が無いため、次の順序で
  F-4 の停止状態へ到達する。
  1. サイクル末尾に、表示中の画像と未キャッシュの最終候補が残っている。
  2. 利用者が入力フォルダから最終候補を外部削除または外部移動する。
  3. `rescan` が消失を検出する前に `advance()` が表示中画像を消費し、窓が空になる。
  4. 続く `refill_window()` が最終候補の消失を検出して `play_order` から除去する。
  5. `cursor == play_order.len()` かつ window 空となるが、境界処理へ移行せず停止する。
- `rescan` が外部消失を先に検出した場合は通常この停止を回避できるが、タイマーの実行順序に
  依存するため、外部削除を「発生しない経路」とは扱わない。
- **外部追加だけでは停止状態を新たに作らない。** 新規画像は `pending` を経て
  `play_order` 末尾へ追加されるため、通常は `cursor` と末尾の間を広げる方向に働く。
  ただし、既に停止状態にある場合に偶然復旧させる可能性があるだけであり、正規の復旧手段とは
  みなさない。
- 一般化すると、**アプリ内削除・デコード失敗／恒久除外・`refill_window()` または
  `rescan` による外部消失処理など、候補列を短縮するすべての経路によって
  `cursor >= play_order.len()` に到達し、かつ window が空**になった場合に停止する。

### 4.3 要件

- **`cursor >= play_order.len()` に到達し window が空になった場合も、停止せず正しく遷移する。**
  - `play_order` が空でなければ、**サイクル境界処理（残窓クリア・`cursor=0`・再シャッフル・
    `cycle_count += 1`）へ移行**してスライドショーを継続する。
- `play_order` が空になったときは、既存の「空フォルダ状態」表示へ移行する。
- 空フォルダ状態への初期化・非同期結果の世代更新は**空状態へ遷移した時に1回だけ**行う。
  空状態が継続しているだけのフレームで、世代更新や同じ初期化を繰り返さない。
- 上記の遷移契機を **`advance()` 経由（`shown > 0`）だけに依存させない**。削除・デコード失敗・
  外部消失で末尾到達した場合にも境界判定が働くようにする。
- 境界判定は、候補列を短縮し得る各処理へ個別に場当たり的に追加するのではなく、
  **候補列変更後または補充前後に共通して適用できる状態遷移**として定義し、新たな除去経路が
  増えても同じ停止を再発させないこと。
- `rescan` と `refill_window()` のどちらが外部消失を先に検出しても、最終的な状態遷移が
  同一になること。

### 4.4 制約・整合

- 前身 の不変条件は維持する：窓は `play_order[cursor..]` の連続ミラー、`current_vram` は
  窓内 `vram_size` 合計と一致、原本は読み取り専用（削除は隔離移動のみ）。
- 「1 サイクルで重複なし」の意味は保つ。境界で再シャッフルする際、直前に表示していたパスが
  新サイクル先頭へ来ない既存の回避（`reshuffle_all_avoiding`）も踏襲する。
- `cycle_count` を二重に進めない（境界処理は 1 回のみ）。
- 外部追加は従来どおり `pending` で書き込み完了を確認してから未表示候補へ追加し、
  F-4 の修復を理由にサイクルカウントや表示済み領域をリセットしない。
- 既に停止状態にあるとき新規画像の追加で偶然表示が再開する挙動に依存しない。

### 4.5 受け入れ基準

1. サイクルの最後に残った 1 枚を削除しても停止せず、再シャッフルして表示が継続する。
2. 最後の 1 枚がデコード連続失敗で恒久除外されても停止せず、継続する。
3. 未キャッシュの最終候補を入力フォルダから外部削除し、`rescan` より先に
   `advance()` → `refill_window()` が実行されても停止せず、サイクル境界へ移行する。
4. 同じ外部削除を `rescan` が先に検出した場合も停止せず、3 と整合する状態へ移行する。
5. 外部追加だけでは `cursor`、`cycle_count`、表示済み領域がリセットされず、追加画像が
   従来どおり未表示候補へ加わる。
6. 全画像が削除・除外されて `play_order` が空になった場合は「空フォルダ状態」へ移行する。
7. アプリ内削除、デコード恒久失敗、`refill_window()` の外部消失、`rescan` の外部消失の
   各境界を検証する自動テストを追加する。

## 5. F-5：上部ステータスバーへ「表示済み」枚数を追加

### 5.1 背景

上部ステータスバー（`app.rs` の `TopBottomPanel`）は現在「未表示」「キャッシュ」「サイクル」を
表示するが、**当該サイクルで既に表示し終えた枚数**を示す指標が無い。利用者がサイクルの進捗
（どこまで見たか）を把握できるよう、表示済み枚数を可視化する。

### 5.2 要件

- 上部ステータスバーに「**表示済み**」枚数を追加する。
- 値は **当該サイクルで advance により表示済みとして消費確定した枚数＝`cursor`**とする。
  画面に一時的に描画されただけの先読み表示は、advance で消費されるまで加算しない。
- 既存の「未表示」「キャッシュ」「サイクル」と並べて表示する。

### 5.3 制約・整合

- 「表示済み」＋「未表示」＝ `play_order.len()`（総数）となる
  （既存定義 `未表示 = play_order.len() - cursor` と整合する）。
- サイクル境界で `cursor = 0` に戻るため、「表示済み」も 0 にリセットされる。
- F-4 の停止是正後も、`cursor` の意味（`advance` のみが進める消費確定枚数）は不変であること。

### 5.4 受け入れ基準

1. 上部バーに「表示済み: N」が表示される。
2. `advance` で送るたびに N が増える（最大 2 ずつ）。
3. サイクル境界で N が 0 に戻る。
4. 「表示済み」＋「未表示」が総数（`play_order.len()`）に一致する。

## 6. F-6：右クリックメニューへ「再生／一時停止」を追加

### 6.1 背景

再生／一時停止は現在、`Space` キーと上部バーのトグルボタン（「⏸ 停止」／「▶ 再生」）でのみ
操作できる。画像を右クリックしたコンテキストメニュー（現状は「削除」「キャンセル」）からは
操作できない。マウス操作中に右クリックのまま再生制御できるようにする。

### 6.2 要件

- 右クリックのコンテキストメニューに「再生／一時停止」の操作を追加する。
- 表示するラベルは現在の再生状態（`playing`）を反映する。上部バーのトグルボタンと同様に、
  **再生中は「一時停止」、停止中は「再生」**を提示する。
- メニューからの再生／一時停止は、`Space` キー・上部バーのボタンと**同一の動作**とする
  （`playing` のトグル、再生再開時の `last_advance` の扱いを含む）。

### 6.3 制約・整合

- **F-2 との整合。** F-2 はメニュー表示中に自動送り（advance）を凍結する。本メニュー項目の
  再生／一時停止操作は、メニュー表示中の外部入力抑制（`menu_open` による `Space`／上部ボタンの
  抑制）に妨げられず実行できること。操作後の状態（playing）は F-2 の凍結／再開と矛盾しないこと。
- メニュー操作後はメニューを閉じる（既存の「削除」「キャンセル」と同様）。
- 既存の誤削除防止（パス基準の `menu_target`・キャンセル挙動）には影響しないこと。
- 停止状態からメニュー内の「再生」を選んだ場合は、上部バーや `Space` から再生した場合と同様に
  `last_advance` を操作時刻へ更新し、設定間隔の全時間が経過してから最初の送りを行う。

### 6.4 受け入れ基準

1. 画像を右クリックすると、メニューに再生状態に応じた「再生」または「一時停止」が表示される。
2. 「一時停止」を選ぶと再生が止まり、「再生」を選ぶと再開する（`Space`／上部ボタンと同一挙動）。
3. 選択後にメニューが閉じる。
4. メニュー内の再生／一時停止操作が、F-2 の自動送り一時停止と矛盾なく動作する。

## 7. F-7：管理画面（フォルダ指定・間隔設定・表示枚数）の追加と状態の永続化

### 7.1 背景

前身 では、画像フォルダは起動時に CLI（標準入力）で指定し、スライドショー間隔
（`ADVANCE_INTERVAL_SECS = 15`）と表示枚数（常に 2 枚）は定数で固定、内部状態はすべて
メモリ管理で永続化しない。アプリ内で設定を変更する手段と、再起動をまたいで続きから
再開する手段が無い。

### 7.2 要件

管理画面（設定画面）を追加し、次を操作できるようにする。

- 画像フォルダの指定／変更。
- スライドショー間隔の設定：**5〜60 秒**。
- 表示画像枚数：**1 枚 または 2 枚**。

起動フロー：CLI（標準入力）でのパス入力は廃し、アプリ内の管理画面でフォルダを指定する。
管理画面は上部バーのボタンから開く（既定の開き方）。

### 7.3 即時反映

- 表示枚数（1/2）と間隔秒数（5〜60）の変更は、**確認ダイアログ無しで直ちに反映**する。
- これらの変更では**サイクルのステータス（`play_order`／`cursor`／`cycle_count`）をクリアしない**。
  枚数変更は表示の見せ方（窓に出す枚数・レイアウト）だけを即時に切り替え、進行中のカウントは
  維持する。
- 2 枚表示中に 1 枚表示へ変更した場合、変更直前の右側画像は**未表示扱いのまま**とする。
  右側画像を表示済みとして `cursor` に加算せず、後続の送りで 1 枚画像として改めて表示する。
  設定変更直前に右側へ見えていた画像が後で単独表示されることは、この仕様上の意図した動作とする。

### 7.4 永続化（ディスク保存・再起動後も継続）

- 次をディスクに保存し、**アプリを閉じて再起動しても継続**する。
  - 設定値：画像フォルダパス、間隔秒数、表示枚数。
  - サイクルのステータス：`play_order`（シャッフル順）、`cursor`、`cycle_count`、
    直近表示パス（新サイクル先頭での連続再表示回避用）
    （必要に応じ `fail_counts` 等）。
- 再起動時、**指定フォルダが保存フォルダと一致**すれば、保存した
  `play_order`／`cursor`／`cycle_count` と設定を復元し、**サイクルの続きから**再開する。
- **画像フォルダを変更した時のみ**、保存内容をクリアし、新フォルダで最初から
  （再スキャン＋新しい `delete/` 準備＋新しいシャッフル）。
- 保存先は実行ファイル隣の状態ファイルを既定とする（変更可）。
- 状態ファイルは外部編集・破損を前提とし、復元した `play_order` を無条件には信用しない。
  復元時に現在の指定フォルダ直下の対応画像一覧と突き合わせ、次を行う。
  - 保存 `cursor` は配列分割に使用する前に `min(saved_cursor, saved_play_order.len())` で
    読み取り範囲だけを安全化する。この値を最終 `cursor` として採用してはならない。
  - 指定フォルダ直下に属さないパス、`delete` 配下、非対応拡張子を除外する。
  - 重複パスを 1 件に正規化する。
  - 保存後に消失したパスを除外し、新規追加パスを未表示側へ加える。
  - 突き合わせ後の最終 `cursor` は、有効と判定された表示済みパスの件数から再計算し、
    F-4 の境界／空状態遷移を適用する。
- 状態保存は一時ファイルを経由して既存ファイルを置換し、失敗時は可能な限り直前の正常ファイルを
  維持する。ただし OS・ファイルシステム障害を含む完全なクラッシュ耐性は保証しない。
- 保存先に書き込み権限が無い場合は app-data 等の書き込み可能な場所へフォールバックし、
  実際に採用した保存先を以後の読み書きで一貫して使用する。

### 7.5 制約・整合

- これは 前身 の「内部状態はメモリで管理し永続化しない」という前提からの**明示的な変更点**である
  （状態／設定ファイルを新設）。
- 表示枚数は `advance` の消費枚数（`shown = window.len().min(枚数)`）・補充の表示スロット数・
  並べ表示レイアウトに関わる。枚数変更時はサイクルのカウントを保ったまま窓の表示枚数を作り直す。
- 間隔は 5〜60 秒に制限（範囲外は丸める）。枚数は {1, 2} のみ。
- 終了中にフォルダ内容が増減していても、再起動後の rescan／補充の既存ロジックで突き合わせる
  （消失は除去、追加は未表示側へ）。
- 復元した `cursor` が、突き合わせ後の `play_order` に対して `cursor >= play_order.len()` と
  なった場合も、F-4 の遷移（サイクル境界／空状態）に従い停止しないこと。

### 7.6 受け入れ基準

1. 管理画面でフォルダ・間隔（5〜60）・枚数（1/2）を設定できる。
2. 間隔・枚数の変更が確認ダイアログ無しで即時反映され、サイクルのステータスがクリアされない。
3. 枚数 1 で 1 枚表示、2 で 2 枚表示になる。
4. 間隔が 5〜60 秒の範囲に制限される。
5. アプリ終了→再起動で、同一フォルダなら前回の `play_order`／`cursor`／`cycle_count` と設定が
   復元され、続きから再生される。
6. フォルダを変更すると、サイクルのカウント等がクリアされ、新フォルダで最初から
   （再スキャン＋新しい `delete/` 準備）になる。
7. 終了中にフォルダ内容が増減していても、再起動後に整合し、停止しない（F-4 と整合）。
8. 状態ファイルにフォルダ外パス・重複・`delete` 配下・非対応拡張子が含まれていても、
   復元候補から除外され、表示・削除対象にならない。
9. 状態保存に失敗してもアプリは継続し、直前の正常な状態ファイルを可能な限り保持する。
10. 同一フォルダを表記違いで再選択してもサイクル状態をリセットしない。
11. 破損した保存値で `cursor > play_order.len()` でもクラッシュせず、有効な表示済み件数から
    `cursor` を再計算して復元する。
12. 2 枚表示中に 1 枚へ変更した場合、変更直前の右側画像は未表示として残り、後で単独表示される。

## 8. F-8：巨大画像を扱わず専用フォルダへ自動退避

### 8.1 背景

巨大画像（高解像度）は 1 枚のデコードだけで数百 ms の停止と数百 MB の RGBA を生む。F-1 はこれを別スレッド化
するが、扱う限りは CPU RAM・VRAM の上限管理が複雑化する（バイト予約・例外枠など）。そこで **v1.0 では
巨大画像をそもそもスライドショーの対象にしない**方針とし、見つけたら専用フォルダへ退避する。

### 8.2 要件

- **寸法が上限を超える画像はスライドショーの対象にしない。** 表示も先読みもしない。
- 対象の巨大画像を見つけたら、入力フォルダ直下に**専用の退避フォルダを作成し、そこへ移動**する
  （`delete` と同様の隔離移動。原本の保全＝移動であり消去ではない）。退避後は `play_order` から除く。
- **判定基準はピクセル寸法**（デコード後 RGBA サイズが効くため、ファイルのバイト数ではなく `幅×高さ`）。
  ヘッダ読み（`image::image_dimensions`）で取得し、フルデコードはしない。
- **閾値は実装側で定める**（利用者からは「サイズは任せる」）。初期値は詳細設計に置き、実測で調整可能とする。
- 退避フォルダ内・`delete` 内はスキャン対象外（既存方針どおりサブフォルダは見ない）。

### 8.3 制約・整合

- 退避は読み取り専用方針の**例外（削除と同系統の移動）**。`rename`＋衝突回避（`name (1).ext`）で安全に行う。
  衝突回避命名は `delete` への移動と共通化し、既存ファイルを上書きしない。
- Undo は提供しない（F-3）。退避を戻したい場合は利用者が退避フォルダから手動で戻す。
- F-8 により、F-1 の **CPU RAM バイト予約・巨大表示画像の例外枠は不要**になる（デコード対象が寸法上限以下に
  収まるため、RAM は「先読み枚数 × 上限サイズ」で有界。詳細設計 §4.1）。
- rescan で後から入った巨大画像も、デコード候補になった時点で判定・退避する。
- **退避移動に失敗しても、通常はスライドショーを停止しない。**
  - 通常の失敗時は画面通知を出さず、対象画像を `cursor` を進めずに未表示候補の末尾へ戻し、他の画像の表示を
    継続する。対象が再び表示候補に選ばれた時点で寸法判定と退避移動を再試行する。失敗直後のフレームで
    同じ画像への無限再試行は行わない。
- **同一画像の退避が累計 5 回失敗した場合**は、**スライドショーのループ処理ごと停止する**（補充・送り・退避処理・
  境界判定・rescan をすべて止める）。この時点で未表示ストックの残りも表示不能な例外ファイルである可能性が
  高く、止めて利用者の手動処理に委ねる方針による。「最後の未表示候補を検出してその場で 5 回」ではなく、
  候補に選ばれるたびに 1 回試行し累計 5 回で停止する。
  - 利用者へ**モーダルダイアログ**を表示し、**具体的なファイル名**（および場所・原因）を示す。
  - 原本は消去せず、対象画像は入力フォルダに残す。`play_order` にも残してよい（ループ停止のためストールしない）。
  - 復帰は明示的な利用者操作のみ：手動でファイルを処理してから、フォルダを選び直すか再起動する。
    `halted` 中は**同一フォルダの再選択でも全リセットして復帰**する。

### 8.4 受け入れ基準

1. 寸法上限を超える画像は表示・先読みされない。
2. 該当画像は専用の退避フォルダへ移動され、`play_order` から除かれる。
3. 退避は移動であり、原本の中身は失われない（消去しない）。退避先で同名衝突しても上書きしない。
4. 退避により残り 1 枚やサイクル末尾に至った場合も、F-4 の遷移で停止しない。
5. 通常の退避失敗では通知を出さず、対象を未表示候補の末尾へ戻してスライドショーを継続する。
6. 同一画像の退避が累計 5 回失敗したらループ処理ごと停止し、**ファイル名を含むモーダルダイアログ**を表示する。
   フォルダ再選択（同一フォルダ可）／再起動で復帰する。
7. `delete` と `oversized` の双方で同じ衝突回避命名を使い、同名ファイルを上書きしない。
